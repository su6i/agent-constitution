---

## CLAUDE.md چیست؟

قبل از اولین prompt، قبل از هر خط کد، Claude یک فایل می‌خواند: **CLAUDE.md**

این فایل را به عنوان بریف اولیه‌ای بدان که Claude برای تمام session از آن پیروی می‌کند — نه به عنوان پیشنهاد، بلکه به عنوان دستورالعمل اصلی.

---

## سه سطح فایل

```
~/.claude/CLAUDE.md          ← Global (برای همه پروژه‌ها)
CLAUDE.md                    ← Project (در root پروژه، در git)
CLAUDE.local.md              ← Local (تنظیمات شخصی، gitignored)
```

**چرا این سه سطح مهم است؟**
- **Global**: قوانینی که در هر پروژه‌ای تکرار می‌کنی
- **Project**: context خاص stack و تیم — با git به اشتراک گذاشته می‌شود
- **Local**: ترجیحات شخصی که نباید با تیم share شود

همه چیز را در یک فایل ریختن = دفن شدن قوانین مهم زیر سروصدا.

---

## چرا اکثر CLAUDE.md کار نمی‌کنند

### مشکل ۱: خیلی طولانی

فایل طولانی توجه Claude را پخش می‌کند. هرچه تعداد خطوط بیشتر باشد، احتمال نادیده گرفتن قوانین مهم‌تر بیشتر است.

**هدف:** زیر ۸۰ خط نگه دار.

### مشکل ۲: محتوای اشتباه

این‌ها **اتلاف جا** هستند:
- `"مثل یک senior engineer رفتار کن"` — هیچ رفتار مشخصی را fix نمی‌کند
- `"گام به گام فکر کن"` — در training مدل هست، تکرار لازم نیست
- `"کد تمیز بنویس"` — معیار مشخصی ندارد

**تست هر خط:** اگر این خط را حذف کنم، Claude اشتباه مشخصی می‌کند؟  
- بله → بماند  
- خیر → حذف شود

### مشکل ۳: بدون اولویت‌بندی

بدون ساختار، Claude نمی‌داند کدام قانون مهم‌تر است. ساختار واضح = اجرای بهتر.

---

## ۵ بخش یک CLAUDE.md موثر

### بخش ۱ — دستورات ضروری

Claude هر session را بدون دانش از نحوه build پروژه شروع می‌کند. بگو دقیقاً چه بنویسد:

```markdown
## Commands
- Build: `npm run build`
- Dev: `npm run dev`
- Test (single): `npm test -- path/to/file`
- Test (all): `npm test`
- Lint: `npm run lint:fix`
- Type check: `npx tsc --noEmit`
```

بدون این بخش: Claude با `npm test` شروع می‌کند در حالی که پروژه روی `pnpm vitest` است — سه turn هدر می‌رود.

---

### بخش ۲ — نقشه معماری

Claude هر session را با **صفر** دانش از codebase شروع می‌کند. نقشه بده:

```markdown
## Architecture
- src/lib/services/ → همه business logic اینجاست
- src/components/   → فقط UI stateless
- src/lib/store/    → global state (Zustand)
- src/app/api/      → API routes، هیچ business logic اینجا نیست
- دسترسی به DB فقط از طریق Server Actions یا API routes
```

مهم‌ترین چیز: **کجا نباید چیزی برود** — این از رایج‌ترین اشتباهات معماری جلوگیری می‌کند.

---

### بخش ۳ — قوانین سخت

مهم‌ترین بخش. هر قانون باید یک اشتباه مشخص را پیشگیری کند:

```markdown
## Rules
- NEVER commit .env files or secrets
- همه async calls باید در try/catch باشند
- فقط functional components — هیچ class component
- پیشوند commit اجباری: feat:, fix:, docs:, refactor:
- هر PR باید `npm run verify` را پاس کند
- فقط static export — بدون SSR (deployed روی S3)
- IMPORTANT: بعد از هر تغییر کد، type check را اجرا کن
```

**نکات:**
- قوانین منفی ("هرگز X نکن") به اندازه قوانین مثبت مهم هستند
- `IMPORTANT` یا `YOU MUST` قبل از قوانین بحرانی تاثیر دارد — اما زیاد استفاده نکن، اثرش کم می‌شود
- **زیر ۱۵ قانون** نگه دار — بیشتر از این، توجه را از قوانین مهم منحرف می‌کند

---

### بخش ۴ — ترجیحات Workflow

جلوگیری از رفتارهای آزاردهنده رایج:

```markdown
## Workflow
- قبل از شروع tasks پیچیده، سوال بپرس
- تغییرات minimal — refactor کد غیرمرتبط ممنوع
- بعد از هر تغییر، tests را اجرا کن و failures را قبل از ادامه fix کن
- commit‌های جداگانه به ازای هر تغییر منطقی، نه یک commit غول‌پیکر
- در صورت تردید بین دو رویکرد، هر دو را توضیح بده و بگذار انتخاب کنم
```

---

### بخش ۵ — چه چیزی را نگذاری

```markdown
## چه چیزی را اضافه نکن:
- دستورات شخصیتی ("مثل senior engineer باش")
- قوانین formatting که linter انجام می‌دهد
- قوانین تکراری بین سطوح مختلف
- چیزهایی که Claude خودش یاد می‌گیرد و در memory ذخیره می‌کند
```

---

## Memory خودکار Claude

Claude یادداشت‌های خودش را اینجا نگه می‌دارد:
```
~/.claude/projects/<project>/memory/
```

برای دیدن آنچه Claude از پروژه‌ات یاد گرفته، این دستور را در session بزن:
```
/memory
```

**نتیجه عملی:** بعد از چند session، Claude احتمالاً بخشی از چیزهایی که می‌خواستی در CLAUDE.md بنویسی را خودش capture کرده. این موارد را دوباره ننویس — جای با ارزش فایل را هدر می‌دهی.

## ۵ خطی که بیشترین تاثیر را دارند

| خط | چه اشتباهی را پیشگیری می‌کند |
|---|---|
| `IMPORTANT: بعد از هر تغییر type check کن` | تحویل کد با broken types |
| `تغییرات minimal — refactor غیرمرتبط ممنوع` | بازنویسی کامل فایل‌های نامرتبط |
| `commit‌های جداگانه، نه یک commit غول‌پیکر` | یک commit با ۴۷ فایل مخلوط |
| `در تردید، هر دو رویکرد را توضیح بده` | تصمیمات معماری که باید مال تو بود |
| `فقط static export — بدون SSR` | اضافه شدن server code به پروژه static |

## اشتباه اساسی

مردم CLAUDE.md را مثل لیست آرزو یا prompt شخصیتی تلقی می‌کنند.

**"باش senior." "باش دقیق." "مثل متخصص فکر کن."**

این brief نیست — این تفکر جادویی است.

- قوانین مشخص و پیشگیرانه
- ترجیحات workflow
